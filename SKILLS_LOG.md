# Skills Log

Running log of tools, languages, and concepts learned while building this project. Feeds resume updates via the InternPilot project — copy new entries over when updating application materials.

Note: this log restarts for the AI PCB Design Review Platform (pivoted 2026-07-13 from a prior FPGA/PCB robotic-arm project — that project's own skills log covered SystemVerilog, Icarus Verilog, and closed-loop control simulation, and remains in this repo's git history).

## Phase 0 — parser + internal board model

- **KiCad `.kicad_pcb` file format**: modern KiCad board files are plain S-expressions (`(token child1 child2 ...)`), while `.kicad_pro` project files are JSON. Learned the grammar (footprints, `fp_text`/`property` refdes and value fields, `segment`/`via`/`zone`/`gr_rect` graphic and copper elements, per-net `(net id "name")` declarations referenced by id elsewhere in the file) well enough to write a parser without needing a KiCad install.
- **Hand-rolled S-expression parsing**: wrote a small tokenizer + recursive-descent reader (`app/parser/sexpr.py`) producing a plain nested-list tree, plus `find_all`/`find_first`/`child_values` helpers for walking KiCad's tag-first-element convention. Chose this over the `pcbnew` Python bindings specifically to avoid requiring a full KiCad desktop install as a runtime dependency — a deliberate "runs anywhere" tradeoff, not just the path of least resistance (`pcbnew` wasn't available on the dev machine either).
- **Pydantic v2 mutable defaults**: confirmed `list[X] = []` field defaults are per-instance (not shared) in Pydantic v2, unlike a plain dataclass — this simplified the `Net.traces`/`Net.vias` accumulation logic in the parser.
- **Test-fixture authoring for a binary-ish/DSL file format**: rather than depending on exporting a real board from a KiCad install, hand-authored a minimal-but-valid `.kicad_pcb` fixture (`examples/simple_board/`) covering every construct the parser needs (board outline, two copper layers, four component kinds, three nets, traces, a via, a copper pour) — deliberately exercises the parser's edge cases rather than being a realistic board.

## Phase 1 — deterministic engineering-check engine

- **PCB engineering heuristics as code**: translated real design-review judgment calls (acute-angle bends causing acid traps, decoupling-cap loop length, daisy-chained vs. planar power distribution, differential-pair skew, thermal relief copper, annular ring / trace-width fab minimums) into explicit, tunable thresholds in each `app/analysis/*.py` module — the kind of heuristics normally kept as tribal knowledge in a senior engineer's head.
- **Geometry from raw coordinates**: implemented endpoint-adjacency detection (shared trace endpoints within a tolerance) and bend-angle computation via the dot-product formula (`cos θ = (v1·v2)/(|v1||v2|)`) to detect acute-angle routing and branch points, without any CAD/geometry library.
- **Net-name convention parsing**: auto-identifying power/ground/clock nets and differential pairs from naming convention alone (`+3V3`, `_P`/`_N`, `NET+`/`NET-`) rather than requiring an explicit schematic-level declaration — a deliberate scope tradeoff, documented inline, since the parser doesn't carry full schematic semantics.
- **Deliberately transparent thresholds over ML**: every check is a named constant (e.g. `LONG_TRACE_WARN_MM = 60.0`) rather than a learned/black-box score — makes "why did this fire" a one-line answer, which matters for the project's "defensible in an interview" goal.
- **Honest scope boundaries**: manufacturability's silkscreen-over-pad / copper-sliver checks from `DESIGN.md`'s catalogue were *not* implemented, because the board model doesn't capture silkscreen text bounding boxes or full pour polygon shape — documented as a gap in the module docstring rather than stubbing a check that could never actually fire on real data.
- **Test design for heuristic checks**: each of the 9 categories gets synthetic-board unit tests (via a small `tests/factories.py` builder module) proving both the positive case (check fires on a deliberately bad board) and the negative case (stays silent on a clean one) — geometry hand-computed (e.g. constructing an exact 30° bend via `cos`/`sin`) rather than eyeballed.

## Phase 2 — scoring + first HTML report

- **Transparent scoring over ML**: implemented the engineering score as a plain, auditable deduction formula (start at 100, subtract `severity_weight × confidence` per issue, floor at 0, overall = average of subscores) rather than any kind of learned model — the explicit goal being that "how was this score computed" has a one-line, defensible answer.
- **Matplotlib in a headless/non-interactive context**: used the `Agg` backend (`matplotlib.use("Agg")` before importing `pyplot`) to render charts to an in-memory buffer with no display/GUI dependency, then embedded them as base64 `data:image/png;base64,...` URIs directly in the HTML — keeps the report a single self-contained file with no external asset references, which matters for emailing it or feeding it into a future PDF export step.
- **XSS-safe HTML generation without a templating engine**: every user/board-derived string (board name, issue text, refdes references) goes through `html.escape()` before interpolation into the report string — verified with a dedicated test that injects `<script>` into an issue summary and asserts it comes out escaped.
- **Visual QA on generated static HTML**: rather than trusting "tests pass" for a human-facing report, served the generated file over a local `http.server` and screenshotted it with Playwright to actually look at spacing, chart rendering, and color-coding before calling the phase done — caught nothing broken this time, but confirmed the practice is worth doing for any future UI-facing output.

## Phase 3 — AI Engineering Review Layer

- **Constraining an LLM to be a narrator, not an analyst**: the entire architecture challenge in this phase was keeping Claude from re-deriving findings the deterministic engine already owns. Solved with three layers, not one: (1) a strict system prompt explicitly forbidding invented findings, (2) an evidence-grounding block (severity counts, highest-impact categories, most common recommendation) computed in Python and handed to Claude pre-digested rather than left for the model to infer, (3) a requirement to cite issue IDs when referencing specific findings. No single one of these is bulletproof; together they make "the AI made something up" a much narrower failure mode, and each is independently explainable in an interview.
- **Designing a data boundary as a real architectural constraint, not a suggestion**: `app/ai/summarizer.py` builds the only object Claude ever sees. Wrote a dedicated test (`test_issue_dicts_carry_id_and_no_raw_geometry`) that asserts `location`/`x`/`y` never appear in the digest — turning "Claude never sees raw geometry" from a design-doc claim into something CI actually enforces.
- **Stable cross-run identifiers for a collection assembled from independent modules**: `Issue.id` (e.g. `PWR-004`) couldn't be assigned inside each `check()` function, because a category's numbering would reset every run in a way that doesn't compose — moved assignment to the orchestrator (`run_all_checks`) instead, one pass after all 9 categories' issues are collected, so IDs stay stable and sequential regardless of which checks fired.
- **Testing an LLM-calling function without an LLM**: no `ANTHROPIC_API_KEY` was available in this dev environment, so `generate_review`/`answer_question` take an optional `client` argument (defaulting to a real `anthropic.Anthropic()` only when unset). Tests inject a small fake client that records the exact `system`/`messages` payload sent — proves the prompt-construction logic is correct without spending a single API call, and the same seam is what a future test suite would use to assert against a real API in CI if desired.
- **Revisiting an already-"done" phase's output**: adding issue IDs made the Phase 2 HTML report's issues table visibly incomplete (no traceability column), so went back and added one rather than treating Phase 2 as frozen — a small reminder that "phase complete" means "the phase's goal was met," not "this file is never touched again."
- **Defense-in-depth over trusting a single prompt instruction**: after external review, added a second layer behind "reference issue IDs" — `find_unsupported_citations()` regex-extracts ID-shaped tokens from Claude's actual output and diffs them against the digest's real IDs, logging a warning if the model cites something that doesn't exist. Same pattern already used for the raw-geometry boundary (test-enforced, not just prompt-enforced) — a prompt is a request, not a guarantee, and the code should be able to notice when the model doesn't comply.
- **Recognizing an unfounded metric before building it**: external feedback suggested surfacing an "estimated review time saved" stat in the report. Declined to implement it as proposed — there's no deterministic basis for that number anywhere in the system, and fabricating one would contradict every other design decision in this project (transparent scoring, documented scope cuts instead of fake checks, no black-box confidence). Worth noting as a judgment call, not just a feature backlog item: not every plausible-sounding feature suggestion survives contact with "can I defend this number in an interview."

## Post-Phase-3 polish (metrics, architecture diagram)

- **Auditing a number before repeating it**: "27 deterministic checks" had been quoted verbatim across several chat summaries (and made it into a draft resume bullet) without ever being recounted after Phase 1 landed. Actually counting `def _check_` across `app/analysis/*.py` found 28, not 27. Small discrepancy, but the fix matters more than the number: don't let a plausible-sounding stat calcify into fact just because it was said with confidence once. `README.md`'s metrics table is now generated from a real grep, not memory.
- **Diagram generation without the intended tool**: the `/diagram` skill's mermaid pipeline depends on a browse-daemon (`$B`) that isn't set up in this environment. Rather than forcing that dependency, built `docs/images/architecture.png` directly with matplotlib (`FancyBboxPatch`/`FancyArrowPatch`) — the same charting library already in `requirements.txt` for the HTML report, so it added zero new dependencies. Good reminder that the "right" tool and the "available" tool aren't always the same, and a hand-rolled fallback using something already in the stack often beats fighting a missing dependency.

## CLI

- **A CLI is a packaging problem before it's a code problem**: `python -m app.cli` works immediately, but "a real `pcbinsight` command" needed an actual entry point — `[project.scripts]` in a new `backend/pyproject.toml`, separate from `requirements.txt` (which stays the pinned source of truth for `pip install -r`). Learned to keep runtime deps in `pyproject.toml`'s `dependencies` and dev-only tools (pytest, httpx) in `[project.optional-dependencies].dev`, rather than one undifferentiated list.
- **Batch processing needs a different failure posture than single-item processing**: `_review_single` lets an exception propagate (crash with a traceback — correct for one explicit target, since silent failure would be worse). `_review_batch` wraps each board's analysis in a broad `except Exception` and continues — deliberately different, because "one bad file kills a 20-board batch run" is a real usability failure a team would hit immediately. Wrote a test that feeds it a deliberately malformed `.kicad_pcb` (unbalanced parens) to prove the rest of the batch still completes.
- **Output-path collisions are a real design constraint, not an edge case**: the obvious first design keys batch report output folders by the *parsed* `board.name` (the `.kicad_pcb` filename stem) — but two board projects can easily share a filename (e.g. every KiCad tutorial board is named `board.kicad_pcb`), which would silently overwrite one report with another. Caught this before writing the implementation and keyed on the *input directory name* instead, which is guaranteed unique among siblings by the filesystem itself; the batch test uses two copies of the same fixture (same internal board name, different folder names) specifically to prove this.

## Phase 4 — core dashboard (FastAPI + React)

- **`Path.name` as a cheap, correct path-traversal guard**: the upload endpoint writes each uploaded file to a temp directory using `Path(upload.filename).name` rather than the raw filename. `.name` always extracts just the final path segment — `Path("../../etc/passwd").name` returns `"passwd"`, not an escaped path — so it's simultaneously the simplest way to flatten a KiCad project's files into one directory *and* a correct defense against a malicious filename, without needing separate sanitization logic. Wrote a test that submits a `"../../evil.kicad_pcb"` filename and asserts it lands harmlessly rather than escaping the temp directory.
- **Removing dead code instead of leaving "just in case" checks**: initially wrote an explicit `if not files: raise HTTPException(400, ...)` guard in the review endpoint. A test proved FastAPI's own request validation already rejects an empty required-file-list with a 422 before the handler body ever runs, making that guard unreachable. Deleted it rather than leaving it as defensive dead code — same "don't handle scenarios that can't happen" discipline applied throughout this project's backend.
- **CORS scoped to the actual local dev origin, not wildcarded**: `allow_origins=["http://localhost:5173", ...]` rather than `"*"`, even though this is explicitly a local-only tool with no public deployment. Good hygiene doesn't become optional just because the stakes are low right now.
- **Verifying a frontend means actually driving it, not just `tsc` passing**: a clean TypeScript build proves the types line up, not that the feature works. Started both the FastAPI and Vite dev servers, then used a Playwright browser to actually drag/upload the real fixture board through the real running backend and screenshot the result — caught nothing broken this time, but it's the only way to know the multipart upload, CORS config, and JSON deserialization all actually agree with each other end-to-end, which no unit test in isolation would catch.
- **A well-designed data model pays off in a later, unplanned way**: the issue browser's expandable "why does this matter" panel needed zero new backend work — `Issue.explanation`/`principle`/`suggested_fix` already existed from Phase 1, so building the Rule Explanation UI stretch goal (discussed several sessions ago) turned out to be a pure frontend task once the dashboard existed to put it in.
